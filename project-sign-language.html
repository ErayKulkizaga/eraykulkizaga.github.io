My Contribution

My primary responsibility in the graduation-project team was model design, training, and evaluation. The surrounding interface, integration, data preparation, and 3D animation work was shared across the team.

  • Designed and trained the recognition-model variants evaluated on the AUTSL dataset.
  • Compared Transformer, BiLSTM, temporal-attention, and class-filtering experiments.
  • Evaluated Top-1, Top-3, and Top-5 performance and investigated weak or visually similar sign classes.
  • Helped select the 179-class MediaPipe/BiLSTM model for the product pipeline; although the later RTMW ensemble achieved higher offline accuracy, the selected model fit the existing live pipeline more reliably.

System Capabilities

Live Recognition

MediaPipe runs in the browser and converts webcam input into hand landmarks. Sixteen-frame sequences are sent to the FastAPI backend through /ws, where the selected BiLSTM runtime returns the top predictions, confidence values, and inference latency.

  • Browser-side MediaPipe Holistic landmark extraction
  • 16-frame sequence collection and WebSocket inference
  • 179-class runtime configuration with Top-3 results
  • Health and startup diagnostics exposed by FastAPI

Dictionary & Avatar

The public demo also exposes sign and landmark endpoints for a Three.js animation experience. Local landmark sequences drive the browser renderer, with smooth transitions between requested words.

  • /signs and /landmark/{word} endpoints
  • Local JSON and BVH animation assets
  • Three.js browser rendering

Runtime Pipeline

1

Browser Landmark Extraction

MediaPipe Holistic runs in browser-side WASM. It extracts left- and right-hand landmarks from webcam frames and starts a sequence only when hands are detected.

2

Sequence Transport

The client collects 16 frames and sends landmark payloads over /ws. The FastAPI service maintains a separate buffer for each WebSocket session.

3

Runtime Preprocessing

The selected model contract normalizes 126 hand-landmark coordinates relative to the wrists, combines them with 30 finger-angle features, and assembles 156 features across a 16-frame temporal sequence.

4

BiLSTM Prediction

The 179-class BiLSTM model with temporal attention returns Top-3 labels, confidence scores, and latency. It was retained because it fits the existing live pipeline without replacing the frontend feature contract.

Model Selection & Trade-off

The highest-accuracy experiment was not automatically the best deployment choice. SignTurk selected the 85.65% Top-1 model because it matched the established MediaPipe/WebSocket pipeline, making the end-to-end system easier to integrate and demonstrate reliably. The 94.09% RTMW ensemble remained an offline research result.

ModelStatus and decision
179-class BiLSTM + temporal attentionSelected product model; 16 frames, 156 features, and 85.65% reported Top-1 accuracy. It aligned with the existing MediaPipe/WebSocket pipeline.
226-class RTMW ensembleAchieved 94.09% Top-1 in offline evaluation, but was not integrated because it required a different extractor and multi-stream runtime.

Engineering decision: optimize for a working end-to-end product contract, then treat higher-accuracy model migration as a separate integration task with explicit preprocessing and regression tests.
